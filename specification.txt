Abstract:
The project aims to design and implement a database-driven system to record, manage, and
evaluate student performance in a course. The system supports dynamic assessment
structures, allowing instructors to add or modify assignments, quizzes, or exams at any point
during the course lifecycle without requiring schema redesign. Each assessment component
can have an associated weight, and the system computes weighted total marks for every
student. Additionally, the system supports flexible grading policies by allowing instructors to
define and update grade cutoffs dynamically. The Oracle DBMS serves as the core of the
system, ensuring data consistency, enforcing integrity constraints, and efficiently performing
aggregate computations required for grading and performance analysis.
Requirement Specifications:
REQ 1: The system shall store records for Students, Courses, Assessment Components, Marks,
and Grading Policies, supporting a variable number of assessments per course without
predefined limits.
REQ 2: The system shall allow instructors to dynamically add, update, or remove assessment
components (e.g., Assignments, Quizzes, Exams) along with their weightages, with changes
reflected automatically in total score computations.
REQ 3: The system shall compute weighted total course marks for each student using SQL
aggregation queries, ensuring that the sum of assessment weights is validated at the database
level.
REQ 4: The system shall support flexible grading by allowing instructors to define grade cutoff
ranges (e.g., A ≥ 85, B ≥ 70), which can be modified without recalculating stored marks.
REQ 5: The system shall assign final grades to students based on computed total marks and
defined cutoffs, implemented using database views or stored procedures.
REQ 6: The system shall enforce data integrity constraints to prevent invalid mark entries (e.g.,
marks exceeding maximum limits) using CHECK constraints, triggers, or stored procedures.
REQ 7: The system shall generate performance reports such as class averages, highest and
lowest scores using SQL aggregate and grouping operations.

CREATE TABLE STUDENT (

CREATE TABLE FACULTY (
    faculty_id NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name VARCHAR2(120) NOT NULL,
    email VARCHAR2(255) NOT NULL UNIQUE,
    password_hash VARCHAR2(255) NOT NULL,
    department VARCHAR2(20) NOT NULL
);

CREATE TABLE COURSE (
    course_id VARCHAR2(20) PRIMARY KEY,
    title VARCHAR2(255) NOT NULL,
    course_type VARCHAR2(20) NOT NULL,
    credits NUMBER(2) NOT NULL,
    department VARCHAR2(20) NOT NULL
);

CREATE TABLE COURSE_OFFERING (
    course_offering_id NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    academic_year NUMBER(4) NOT NULL,
    semester NUMBER(2) NOT NULL,
    status VARCHAR2(20) DEFAULT 'current' NOT NULL,

    course_id VARCHAR2(20) NOT NULL,

CREATE TABLE ASSESSMENT (
    assessment_id NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    assessment_type VARCHAR2(60) NOT NULL,
    total_marks NUMBER(4) NOT NULL,

    assessment_date DATE DEFAULT SYSDATE NOT NULL,
    course_offering_id NUMBER NOT NULL,

      REFERENCES course_offering(course_offering_id)

    min_marks NUMBER(5,2) NOT NULL,

    course_offering_id NUMBER NOT NULL,

      REFERENCES course_offering(course_offering_id)

    CONSTRAINT fk_sgpa_student FOREIGN KEY (student_id) REFERENCES student(email)

CREATE TABLE ATTEMPTS (

1.	Compute Total Weighted Marks
SELECT
    a.studentMail,
    SUM((a.marksObtained / b.totalMarks) * b.weight) AS total_score
FROM ATTEMPTS a
JOIN ASSESSMENT b ON a.assessmentID = b.assessmentID
GROUP BY a.studentMail;
2.	Assign Grades

SELECT
    t.studentMail,
    t.total_score,
    g.grade
FROM (
    SELECT
        a.studentMail,
        SUM((a.marksObtained / b.totalMarks) * b.weight) AS total_score
    FROM ATTEMPTS a
    JOIN ASSESSMENT b ON a.assessmentID = b.assessmentID
    GROUP BY a.studentMail
) t
JOIN GRADE_CUTOFFS g
ON t.total_score BETWEEN g.minMarks AND g.maxMarks;


Compute weighted total + validate weights

CREATE TRIGGER check_weight
BEFORE INSERT ON ASSESSMENT
FOR EACH ROW
BEGIN
DECLARE total INT;

SELECT IFNULL(SUM(weight),0) INTO total
FROM ASSESSMENT
WHERE courseOfferingID = NEW.courseOfferingID;

IF total + NEW.weight > 100 THEN
SIGNAL SQLSTATE '45000'
SET MESSAGE_TEXT = 'Total weight exceeds 100';
END IF;
END;

4. TRIGGERS

•	Trigger: Prevent invalid marks

CREATE OR REPLACE TRIGGER check_marks
BEFORE INSERT ON ATTEMPTS
FOR EACH ROW
DECLARE
    maxMarks NUMBER;
BEGIN
    SELECT totalMarks INTO maxMarks
    FROM ASSESSMENT
    WHERE assessmentID = :NEW.assessmentID;

    IF :NEW.marksObtained > maxMarks THEN
        RAISE_APPLICATION_ERROR(-20001, 'Marks exceed maximum limit');
    END IF;
END;

Procedure: Get Student Total Marks

	CREATE OR REPLACE PROCEDURE GetTotalMarks(p_student VARCHAR2)
IS
total_score NUMBER;
BEGIN
SELECT SUM((a.marksObtained / b.totalMarks) * b.weight)
INTO total_score
FROM ATTEMPTS a
JOIN ASSESSMENT b ON a.assessmentID = b.assessmentID
WHERE a.studentMail = p_student;

DBMS_OUTPUT.PUT_LINE('Total Score: ' || total_score);
END;

Class Report Procedure

CREATE OR REPLACE PROCEDURE ClassReport
IS
    avg_marks NUMBER;
    high NUMBER;
    low NUMBER;
BEGIN
    SELECT AVG(marksObtained), MAX(marksObtained), MIN(marksObtained)
    INTO avg_marks, high, low

    DBMS_OUTPUT.PUT_LINE('Highest: ' || high);
    DBMS_OUTPUT.PUT_LINE('Lowest: ' || low);
END;

Get Grade Function

CREATE OR REPLACE FUNCTION GetGrade(
score NUMBER,
cid NUMBER
) RETURN CHAR
IS
g CHAR(2);
BEGIN
SELECT grade INTO g
FROM GRADE_CUTOFFS
WHERE score BETWEEN minMarks AND maxMarks
AND courseOfferingID = cid
AND ROWNUM = 1;

RETURN g;
END;

iii. Discuss design alternatives and normal form of the derived schema in detail. In case the implementation follows a different schema highlight it clearly here with necessary justifications


Design Alternatives

1. Redundancy

Alternative Design (High Redundancy)
•	Store multiple derived aggregates (total marks, weighted score, grade) in separate result tables.
•	Keep duplicated course/faculty/student data in transaction rows.

Current Design (Mostly Normalized, Controlled Denormalization)
•	Weighted totals are still computed from ATTEMPTS + ASSESSMENT when needed.
•	final_grade is stored in ATTENDS as a controlled denormalized field for fast retrieval and stable display.
•	This stored grade is recomputed on marks updates and cutoff updates through backend logic.

Justification
•	Most schema remains normalized.
•	One intentional denormalized attribute (final_grade) improves read performance and UX while preserving consistency through recomputation.

2. Incompleteness

Alternative Design (Incomplete)
•	Missing offering lifecycle/status tracking.
•	Missing persisted final grade state per enrollment.

Current Design (Complete for Implemented Features)
Includes:
•	STUDENT, FACULTY, COURSE, COURSE_OFFERING
•	ASSESSMENT, ATTEMPTS, ATTENDS
•	GRADE_CUTOFFS, SGPA
•	Offering status in COURSE_OFFERING.status
•	Stored enrollment grade in ATTENDS.final_grade

Justification
•	The schema supports current vs archived offerings, incremental assessment completion, dynamic cutoff updates, and persisted final grade outcomes.

3. Decomposition

Poor Design (No Decomposition)
STUDENT_COURSE_DATA(
    studentID,
    courseID,
    facultyName,
    assessmentType,
    marks,
    grade
)

Current Design (Proper Decomposition)
Split across normalized relations:
•	STUDENT
•	FACULTY
•	COURSE
•	COURSE_OFFERING
•	ASSESSMENT
•	ATTEMPTS
•	ATTENDS
•	GRADE_CUTOFFS
•	SGPA

Justification
•	This decomposition reduces duplication and keeps transactional updates local to relevant entities.

Normalization of Derived Schema

🔹 First Normal Form (1NF)
 Condition:
•	All attributes are atomic
•	No repeating groups or multivalued attributes

 Verification:
•	Each column contains a single value
•	No repeating columns like Quiz1, Quiz2
 Example:
marksObtained = single value

 Conclusion:
 Schema is in 1NF

🔹 Second Normal Form (2NF)
 Condition:
•	Must be in 1NF
•	No partial dependency on composite keys

 Verification:
Example:
ATTEMPTS(student_mail, assessment_id → marks_obtained, attempt_date)
•	marks depends on full composite key
ATTENDS(student_mail, course_offering_id → status, enrollment_date, final_grade)
•	fully dependent

 Conclusion:
 Schema is in 2NF

🔹 Third Normal Form (3NF)
 Condition:
•	Must be in 2NF
•	No transitive dependency

 Verification:
•	COURSE attributes are separated from COURSE_OFFERING.
•	ASSESSMENT attributes are separated from ATTEMPTS.
•	GRADE_CUTOFFS is separated from ATTENDS/ATTEMPTS.
•	No non-key attribute in core transactional tables depends on another non-key attribute in the same table.

 Conclusion:
 Core schema is in 3NF, with one intentional denormalized field: ATTENDS.final_grade.

🔹 Boyce-Codd Normal Form (BCNF)
       Condition:
       For every functional dependency:
          X → Y
       X must be a super key

 Verification:
•	assessment_id → assessment_type, total_marks, weight, assessment_date, course_offering_id
•	course_offering_id → academic_year, semester, status, branch, course_id, faculty_id
•	(student_mail, assessment_id) → attempt_date, marks_obtained
•	(student_mail, course_offering_id) → enrollment_date, status, final_grade
Key-based dependencies are preserved in core relations.

 Conclusion:
 Core transactional design is BCNF-oriented; ATTENDS.final_grade remains a controlled denormalization for performance and product behavior.


Implementation Differences from Initial Design

 Initial Mapping
•	Separate relationship tables:
o	HAS
o	DEFINES
o	TEACHES

 Final Implementation
•	Relationships merged using foreign keys:
Relationship	Implemented As
HAS	courseOfferingID in ASSESSMENT
DEFINES	courseOfferingID in GRADE_CUTOFFS
TEACHES	facultyID in COURSE_OFFERING

Additional Implemented Differences
•	COURSE_OFFERING includes status to model current vs archived offerings.
•	ATTENDS includes final_grade as persisted computed outcome.
•	Naming was standardized to snake_case in implemented schema.

 Justification
Many-to-one relationships with total participation are merged into entity tables to eliminate redundancy and simplify the schema without losing information. Controlled denormalization (ATTENDS.final_grade) is maintained by recomputation logic after marks/cutoff updates.

 Final Conclusion
The database schema is designed to minimize redundancy, avoid incompleteness, and ensure proper decomposition. The core transactional model satisfies 1NF, 2NF, 3NF, and BCNF principles, with one intentional denormalized field (ATTENDS.final_grade) to support efficient grade display and stable results. The final implementation preserves ER semantics while adapting the schema for production behavior and performance.
